## 新增需求

### 需求:续办触发判断
系统必须在每日续办检查任务中，对每个启用了自动续办的车牌，通过 stateList 接口判断是否需要续办。当六环外进京证（`jjzzl=02`）的 `bzxx` 中最新记录剩余有效天数为 1 天（即明天到期），且该车辆 `ecbzxx` 为空（无待审记录），且 `elzsfkb=true`（六环外可办）时，必须触发自动续办流程。

#### 场景:证件明天到期且无待审记录
- **当** 车辆最新六环外进京证的 `yxqz` 为明天的日期，且 `ecbzxx` 为空，且 `elzsfkb=true`
- **那么** 系统触发该车牌的自动续办流程

#### 场景:证件已过期且无待审记录
- **当** 车辆最新六环外进京证的 `yxqz` 早于今天，且 `ecbzxx` 为空，且 `elzsfkb=true`
- **那么** 系统触发该车牌的自动续办流程

#### 场景:已有待审记录
- **当** 车辆的 `sfyecbzxx=true` 或 `ecbzxx` 非空
- **那么** 系统必须跳过该车牌的续办，记录日志说明"已有待审记录，跳过续办"

#### 场景:六环外不可办理
- **当** 车辆的 `elzsfkb=false`
- **那么** 系统必须跳过续办，并通过该车牌配置的通知渠道推送"六环外进京证当前不可办理"的告警

#### 场景:证件有效期充足
- **当** 车辆最新六环外进京证的剩余有效天数大于 1 天
- **那么** 系统必须跳过续办，不执行任何操作

### 需求:续办随机时间调度
系统必须在配置的时间窗口内随机选择一个时刻执行续办操作，避免每日在固定时间发起请求。

#### 场景:在时间窗口内随机执行
- **当** 续办检查任务被触发
- **那么** 系统必须在 `time_window_start` 到 `time_window_end` 之间随机选择一个时刻，通过异步延迟等待到该时刻后再执行续办 API 调用链

#### 场景:使用默认时间窗口
- **当** 用户未配置 `global.auto_renew.time_window_start` 和 `time_window_end`
- **那么** 系统必须使用默认时间窗口 00:00 至 06:00

### 需求:续办API调用链
系统必须按照以下顺序依次调用续办相关 API，每步均须校验 `code==200`，任何步骤失败必须中断整个流程并通知用户。

#### 场景:完整调用链成功
- **当** 触发续办流程
- **那么** 系统必须依次调用以下 API 并校验每步成功：
  1. `applyVehicleCheck` — 提交 `{hphm, hpzl}`，校验车辆
  2. `getJsrxx` — 提交 `{}`，获取驾驶人信息 `{jsrxm, jszh, dabh}`
  3. `applyCheckNum` — 提交驾驶人信息，校验驾驶人
  4. `checkHandle` — 提交 `{vId, jjzzl:"02", hphm}`，获取可选进京日期 `jjrqs`
  5. `checkInputRoadInfo` — 提交 `{vId}`，检查是否需填行驶路线
  6. `insertApplyRecord` — 提交完整申请数据

#### 场景:选择最早可用日期
- **当** `checkHandle` 返回 `jjrqs` 数组
- **那么** 系统必须选择数组中的第一个日期作为 `jjrq`（进京日期），以实现证件衔接无空窗期

#### 场景:无可选日期
- **当** `checkHandle` 返回的 `jjrqs` 为空数组
- **那么** 系统必须中断续办流程并通知用户"当前无可选进京日期"

#### 场景:校验步骤失败
- **当** 调用链中任何步骤返回 `code != 200`
- **那么** 系统必须立即中断续办流程，记录失败步骤名称和完整响应体，并通过该车牌配置的通知渠道推送错误信息

### 需求:续办申请数据组装
系统必须从多个数据源自动组装 `insertApplyRecord` 所需的完整请求体，禁止缺少任何必需字段。

#### 场景:正确组装请求体
- **当** 所有前置步骤成功完成
- **那么** 系统必须组装请求体，其中：
  - `vId`, `hphm`, `hpzl`, `ylzsfkb`, `elzsfkb`, `elzqyms`, `ylzqyms`, `elzmc`, `ylzmc`, `cllx` 来自 stateList 响应
  - `jsrxm`, `jszh`, `dabh` 来自 getJsrxx 响应
  - `jjrq` 来自 checkHandle 响应的 `jjrqs[0]`
  - `jjzzl` 固定为 `"02"`（六环外）
  - `jjmd`, `jjmdmc`, `area`, `jjdq`, `xxdz`, `jjdzgdjd`, `jjdzgdwd`, `sfzj`, `zjxxdz`, `zjxxdzgdjd`, `zjxxdzgdwd`, `sqdzgdjd`, `sqdzgdwd` 来自用户配置
  - `txrxx` 固定为 `[]`，`jingState`, `jjlk`, `jjlkmc`, `jjlkgdjd`, `jjlkgdwd` 固定为 `""`

### 需求:续办结果通知
系统必须在续办流程结束后（无论成功或失败），通过该车牌已配置的通知渠道推送结果。

#### 场景:续办提交成功
- **当** `insertApplyRecord` 返回 `code==200`
- **那么** 系统必须推送通知，内容包含：车牌号、进京日期、进京证类型（六环外）、"已提交，等待审核"

#### 场景:续办失败
- **当** 续办流程中任何步骤失败
- **那么** 系统必须推送通知，内容包含：车牌号、失败步骤名称、错误原因摘要

#### 场景:Token失效
- **当** 续办过程中 API 返回 Token 相关错误
- **那么** 系统必须推送通知，明确提示用户"Token 已失效，请手动更新配置中的 token"

### 需求:防重复提交
系统必须防止同一车牌在同一天内重复提交续办申请。

#### 场景:当日已提交过续办
- **当** 系统在 Redis 中检测到该车牌当日已有续办提交记录
- **那么** 系统必须跳过续办，记录日志说明"当日已提交续办，跳过"

#### 场景:首次提交续办
- **当** Redis 中无该车牌当日续办记录，且 stateList 确认无待审记录
- **那么** 系统在成功调用 `insertApplyRecord` 后，必须在 Redis 中写入当日续办记录（TTL 24 小时）
